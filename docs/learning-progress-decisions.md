# 学习进度与基础配置 — 设计决策

> 在做「学习进度 / 单词掌握」功能之前先立的规矩。配套接口见 [api.md](api.md) 的
> [Learning settings](api.md#learning-settings) 与 `PUT /api/v1/me/learning-settings`。
> 背景:用户引导时要选 CEFR 等级(A1–C2)与英/美式(BrE/AmE),这两项会被改动,
> 必须先定清楚「改设置对已有学习进度的影响」,否则未来的进度表会被设计带偏。

---

## 0. 现状

截至本文,仓库只有 `auth / user / otp / session`,**尚无任何学习进度 / 单词模块**。
所以这是一份**前瞻决策**:此刻没有代码可改,目标是把原则定死,让将来的进度功能不被
基础配置卡住。

---

## 1. 核心原则:进度按「单词」记,不按「等级」记

> `cefr_level` 只是「当前学习范围的指针」,**绝不作为进度的归属维度**。

进度永远挂在词条(`word_id`)上,与用户当时是什么等级无关。只要守住这条,
「改设置影响进度吗?」就有了干净答案:**不破坏任何已掌握记录,只重新圈定要学的范围。**

---

## 2. 两个设置的影响,完全不同

### 2.1 英式 / 美式(EnglishVariant)—— 纯展示层,零影响

- `colour / color` 是**同一个词条**,只是拼写与发音音频不同。
- 切换行为:已学 / 在学的词**立即**按新拼写渲染、改取新变体音频。
- **不重置、不提示、不需要确认弹窗。**
- 边角:个别词的「拼写记忆」与拼写绑定 → 忽略,掌握度记在词条上,拼写属渲染细节。

### 2.2 CEFR 等级(CEFRLevel)—— 内容范围,重组队列,保留掌握

等级决定「学哪批词池」,所以改它有真实后果,但**后果只落在「要学队列」,不落在「已掌握记录」**:

| 场景 | 已掌握的词 | 学习队列 |
|---|---|---|
| 升级 B1→B2 | 全部保留 | 加入 B2 新词;队列按「新池 − 已掌握」重算 |
| 降级 B2→B1 | 全部保留 | B1 池里已会的自动算掌握,只补没学的 |

- 改等级时**不碰任何进度表**,只让「学习队列服务」在下次取数时按新 `cefr_level` 懒重算。
- 前端给一句**软提示**(不拦截):"已掌握的单词会保留,学习队列将按新等级重新组织"。
  变体切换则不提示。

---

## 3. 现在就要锁的约束

1. **未来的进度表必须按词建模**,例如:

   ```sql
   -- 主键 (user_id, word_id);不要出现 level 列,也不要把 level 拼进任何进度主键
   CREATE TABLE word_progress (
       user_id     UUID NOT NULL REFERENCES users (id) ON DELETE CASCADE,
       word_id     UUID NOT NULL,
       state       TEXT NOT NULL,           -- new / learning / mastered ...
       mastered_at TIMESTAMPTZ,
       updated_at  TIMESTAMPTZ NOT NULL DEFAULT now(),
       PRIMARY KEY (user_id, word_id)
   );
   ```

   `cefr_level` 继续留在 `student_profiles` 当**范围指针**。

2. **`PUT /api/v1/me/learning-settings` 永远保持「进度无关」**。它现在只写 `student_profiles`
   两列,正好符合 —— **别**在这个接口里加任何「清进度 / 迁移进度」逻辑。要重算队列,走领域
   事件或队列服务懒重算,不在写设置时同步做。

3. **`grade` 与 `cefr_level` 的重叠**一并纳入本决策:进度既不挂 `level` 也不挂 `grade`,
   这两者都只是范围维度,谁作废都不影响进度。
